================================================================================
  KEY PATTERNS SUMMARY
================================================================================

PATTERN 1: Function Naming
  Structure: [verb][Resource]([params])
  Examples: getCustomAgents(), createCustomAgent(agent), updateCustomAgent(id, updates)

PATTERN 2: Request Function
  request<ResponseType>(endpoint, {
    method: 'GET|POST|PUT|DELETE',
    body: { /* payload */ }
  })

PATTERN 3: Error Handling
  - Standard endpoints: request<T>() handles automatically
  - Streaming endpoints: Custom fetch with manual error checking
  - Format: "Error message (HTTP 404)"

PATTERN 4: Type Safety
  - Always use generics: request<T>()
  - Use Omit<> for create operations
  - Use Partial<> for updates
  - Use discriminated unions for complex responses

PATTERN 5: Request/Response
  - GET: No body
  - POST: Full data (new fields only)
  - PUT: Partial data (only changed fields)
  - DELETE: No body
  - Query params: ?key=value appended to URL

================================================================================
  COMMON MISTAKES (FROM ANALYSIS)
================================================================================

1. ❌ Using 'any' instead of proper types
   ✓ Always specify: request<ResponseType>()

2. ❌ Missing generic type parameter
   ✓ Always use: request<T>(endpoint)

3. ❌ Using custom fetch for JSON responses
   ✓ Use request<T>() unless streaming

4. ❌ camelCase in request body for backend
   ✓ Use snake_case: api_key, user_id

5. ❌ Not checking discriminated union correctly
   ✓ Check discriminant first: if (!result.valid)

6. ❌ Forgetting error handling for streams
   ✓ Check response.ok manually for custom fetch

7. ❌ Mixing patterns in same codebase
   ✓ Follow established patterns consistently

8. ❌ Not using Omit<> for create operations
   ✓ Omit auto-generated fields: id, user_id, timestamps
