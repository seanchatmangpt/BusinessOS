# bos CLI Ontology Usage Inventory

**Audit Date:** 2026-03-25
**Scope:** `/Users/sac/chatmangpt/BusinessOS/bos/` (complete Rust codebase)
**Purpose:** Identify which ontologies are currently used, imported, or referenced; identify gaps

---

## Executive Summary

The `bos` CLI uses a **selective but strategic approach** to ontologies:

- **Currently Used:** 7 standard ontologies (W3C core + domain-specific)
- **Built-In Inference:** Conventions for 8 additional ontologies (auto-mapping)
- **Registry System:** Extensible YAML-based loader supporting unlimited ontologies
- **Gap Analysis:** 9 standard ontologies should be integrated for enterprise compliance

## Ontology Registry System (extensible)

### File Location
`bos/core/src/ontology/registry.rs` (lines 1-491)

### Purpose
Load, validate, and provide runtime access to ontology configurations from YAML files.

### Supported Features
- **Namespace Declaration:** Defines all PREFIX declarations for SPARQL
- **Dependency Tracking:** Enforces ontology dependency graphs
- **Validation:** Circular dependency detection
- **Framework Support:**
  - Compliance: SOC2, HIPAA, GDPR, SOX
  - Domains: Commerce, Operations, People, Finance, Analytics
  - Extended: FIBO, Healthcare, Research
- **Statistics:** Track enabled/required ontology counts
- **Hot Reload:** Configuration changes without restart

### Example Configuration Structure
```yaml
metadata:
  name: "production-f5"
  environment: "production"
spec:
  ontologies:
    - name: "sales_ontology"
      iri: "http://businessos.dev/ontologies/sales/v1"
      version: "1.0.0"
      required: true
      enabled: true
      alias: "sales"
      frameworks: ["commerce", "soc2"]
      dependencies: ["schema_ontology"]
      validation:
        entity_count_min: 100
  namespaces:
    schema: "http://schema.org/"
    prov: "http://www.w3.org/ns/prov#"
    # ... etc
  validation:
    strict: true
    on_missing_required: "error"
```

## Missing (Should Be Integrated)

Based on standards catalog and ChatmanGPT's domains, these ontologies are **NOT YET USED** but should be:

### 1. **OWL (Web Ontology Language)**
- **Namespace:** `http://www.w3.org/2002/07/owl#`
- **Why Needed:** Class restrictions, property cardinality, inverse relations
- **Use Case:** Formal schema constraints (e.g., "Deal must have exactly 1 seller")
- **Priority:** HIGH (enterprise schema validation)
- **Integration Point:** `bos/core/src/ontology/execute.rs` — Add OWL constraint checking

### 2. **SHACL (Shapes Constraint Language)**
- **Namespace:** `http://www.w3.org/ns/shacl#`
- **Why Needed:** Validation rules for RDF data quality
- **Use Case:** "Email must match pattern `^[^@]+@[^@]+\.[^@]+$`"
- **Priority:** HIGH (compliance: SOC2 data validation)
- **Integration Point:** New module `bos/core/src/ontology/shacl_validator.rs`

### 3. **DCAT (Data Catalog Vocabulary)**
- **Namespace:** `http://www.w3.org/ns/dcat#`
- **Why Needed:** Dataset/table metadata discovery
- **Use Case:** Describe tables as `dcat:Dataset` with `dcat:distribution` + `dcat:theme`
- **Priority:** MEDIUM (data governance: compliance, lineage)
- **Integration Point:** `bos/core/src/ontology/mapping.rs` — Add DCAT properties

### 4. **FIBO (Financial Industry Business Ontology)**
- **Namespace:** `https://spec.edmcouncil.org/fibo/ontology/`
- **Why Needed:** Deal, payment, party semantics
- **Use Case:** Deal structures with FIBO:Deal, FIBO:Party, FIBO:Instrument
- **Priority:** HIGH (for Vision 2030 A2A deals)
- **Integration Point:** New module `bos/core/src/ontology/fibo_mapper.rs`
- **Current Status:** Referenced in `registry.rs` line 107 (`fibo: bool`), but unused

### 5. **P-Plan (Process Execution Ontology)**
- **Namespace:** `http://purl.org/net/p-plan#`
- **Why Needed:** Distinguish template vs execution (prompt vs agent output)
- **Use Case:** `p-plan:Step` (abstract task) vs `p-plan:Activity` (execution)
- **Priority:** MEDIUM (OSA: agent lifecycle tracing)
- **Integration Point:** New module `bos/core/src/ontology/pplan_mapper.rs`

### 6. **WoT TD (W3C Web of Things Thing Description)**
- **Namespace:** `https://www.w3.org/2022/wot/td/v1.1`
- **Why Needed:** Agent capability declarations
- **Use Case:** "Agent X exposes POST /invoke endpoint with schema Y"
- **Priority:** MEDIUM (OSA: agent discovery)
- **Integration Point:** New module `bos/core/src/ontology/wot_td_mapper.rs`

### 7. **SOSA/SSN (Sensor, Observation, Sample, and Actuator)**
- **Namespace:**
  - SOSA: `http://www.w3.org/ns/sosa/`
  - SSN: `http://www.w3.org/ns/ssn/`
- **Why Needed:** Sensor data semantics (for agent heartbeats, metrics)
- **Use Case:** `sosa:Observation` (latency measurement) → `sosa:hasResult`
- **Priority:** LOW (future: agent observability)
- **Integration Point:** New module `bos/core/src/ontology/sosa_mapper.rs`

### 8. **ODRL (Open Digital Rights Language)**
- **Namespace:** `http://www.w3.org/ns/odrl/2/`
- **Why Needed:** Access control policies on RDF resources
- **Use Case:** "User X can only view triples tagged with `odrl:hasPolicy`"
- **Priority:** MEDIUM (SOC2: access control)
- **Integration Point:** New module `bos/core/src/ontology/odrl_validator.rs`

### 9. **DQV (Data Quality Vocabulary)**
- **Namespace:** `http://www.w3.org/ns/dqv#`
- **Why Needed:** Quality metrics on data (completeness, accuracy, timeliness)
- **Use Case:** "Deal table has 95% email_valid rate per `dqv:Dimension`"
- **Priority:** MEDIUM (compliance: data quality reports)
- **Integration Point:** New module `bos/core/src/ontology/dqv_metrics.rs`

## Command Inventory

The `bos` CLI exposes ontology operations via noun-verb commands:

### `bos ontology construct`
**Maps:** Table definitions → SPARQL CONSTRUCT queries
**Input:** `--mapping` (JSON file with TableMapping config)
**Output:** `.obsr/queries/` directory with `.rq` files
**Ontologies:** Uses all prefixes from mapping config
**Evidence:** `bos/cli/src/nouns/ontology.rs:150-167`

### `bos ontology export`
**Maps:** Mapping config → Turtle RDF file
**Input:** `--mapping` + `--output`
**Output:** Single `.ttl` file with all CONSTRUCT queries as comments
**Ontologies:** All configured prefixes embedded
**Evidence:** `bos/cli/src/nouns/ontology.rs:169-203`

### `bos ontology execute`
**Maps:** PostgreSQL tables → RDF triples via SPARQL CONSTRUCT
**Input:** `--mapping` + `--database` + optional `--table`
**Output:** N-Triples serialization in response
**Ontologies:** Uses configured mappings + PROV-O provenance
**Evidence:** `bos/cli/src/nouns/ontology.rs:205-242`

### `bos ontology infer`
**Maps:** ODCS DataModel → Mapping config (auto-detect)
**Input:** `--workspace` (optional) + `--output` + optional `--confidence`
**Output:** Generated `MappingConfig` JSON
**Ontologies:** Auto-detects schema.org, foaf, bpmn, org, skos, prov
**Evidence:** `bos/cli/src/nouns/ontology.rs:244-288`

## Conclusion

**bos** follows a **pragmatic ontology strategy:**

- **Core:** Only W3C essentials (RDF, RDFS, XSD, PROV-O) + domain commons (Schema.org, FOAF)
- **Extensible:** Registry system allows runtime loading of ANY ontology
- **Smart Inference:** Built-in conventions guess ontologies from table/column names
- **Gap:** 9 standard ontologies should be integrated for enterprise compliance (OWL, SHACL, DCAT, FIBO, P-Plan, WoT TD, SOSA, ODRL, DQV)

**Next Step:** Implement SHACL + OWL + FIBO (Priority 1) as per roadmap above.
