# BusinessOS Database Schema Reference

**Version:** 1.0
**Last Updated:** 2026-03-26
**Database:** PostgreSQL 15+
**Compliance Frameworks:** SOC2, GDPR, HIPAA, SOX

---

## Overview

BusinessOS implements a Fortune 5-grade schema supporting:
- **FIBO (Financial Business Ontology)** deals and compliance
- **Data Mesh** with lineage tracking and quality metrics
- **Healthcare PHI (Protected Health Information)** per HIPAA
- **Comprehensive Audit Logging** per SOX (7-year retention)
- **Multi-Framework Compliance Controls** (SOC2, GDPR, HIPAA, SOX)

## Table Reference

### 1. FIBO Deals (Finance Domain)

#### `deals` Table

**Purpose:** Core financial instrument deals with compliance tracking

**Columns:**

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | UUID | PRIMARY KEY | Auto-generated by `gen_random_uuid()` |
| `name` | VARCHAR(500) | NOT NULL | Deal name/identifier |
| `amount_cents` | BIGINT | NOT NULL, > 0 | Amount in cents (avoids float precision) |
| `currency` | VARCHAR(3) | NOT NULL, DEFAULT 'USD' | ISO 4217 code (USD, EUR, GBP, etc.) |
| `status` | VARCHAR(50) | NOT NULL, CHECK | draft, proposed, negotiating, approved, executed, settled, closed, rejected |
| `domain` | VARCHAR(100) | NOT NULL, CHECK | equity, fixed_income, derivatives, commodities, fx, structured |
| `created_by` | UUID | NOT NULL, FK(users) | User who created the deal |
| `assigned_to` | UUID | FK(users), nullable | User responsible for deal |
| `kcy_verified` | BOOLEAN | DEFAULT false | Know Your Customer verification |
| `aml_verified` | BOOLEAN | DEFAULT false | Anti-Money Laundering verification |
| `sox_verified` | BOOLEAN | DEFAULT false | Sarbanes-Oxley compliance |
| `counterparty_id` | UUID | nullable | Related counterparty entity |
| `counterparty_name` | VARCHAR(500) | nullable | Counterparty name (cached) |
| `deal_date` | DATE | nullable | Date deal was struck |
| `settlement_date` | DATE | nullable | Date deal settles |
| `maturity_date` | DATE | nullable | Date deal matures |
| `description` | TEXT | nullable | Deal narrative |
| `risk_rating` | VARCHAR(10) | CHECK, nullable | S&P/Moody's equivalent (AAA-D) |
| `internal_reference` | VARCHAR(100) | nullable | Internal deal ID |
| `external_reference` | VARCHAR(100) | nullable | External/counterparty reference |
| `created_at` | TIMESTAMP TZ | DEFAULT NOW() | Creation timestamp |
| `updated_at` | TIMESTAMP TZ | DEFAULT NOW() | Last update (auto via trigger) |

**Indexes:**
```
idx_deals_status                 — Filter by deal status
idx_deals_created_by             — Find deals by creator
idx_deals_assigned_to            — Find deals by assignee
idx_deals_created_at             — Time-based queries
idx_deals_domain                 — Filter by instrument type
idx_deals_currency               — Currency grouping
idx_deals_deal_date              — Date-based analytics
idx_deals_settlement_date        — Settlement timeline
idx_deals_kcy_aml_sox            — Compliance status checks
idx_deals_counterparty           — Counterparty lookup
```

**Constraints:**
- `amount_cents > 0` — Non-negative amounts
- `deal_date <= settlement_date` — Logical ordering
- `settlement_date <= maturity_date` — Logical ordering
- Foreign key constraint on `created_by` with `ON DELETE RESTRICT`
- Soft foreign key on `assigned_to` with `ON DELETE SET NULL`

**Trigger:**
- `trigger_deals_updated_at` — Auto-updates `updated_at` on modification

**Usage:**
```sql
-- Find all approved deals by user
SELECT * FROM deals
WHERE status = 'approved' AND created_by = $1
ORDER BY created_at DESC;

-- Check compliance status
SELECT id, name, kcy_verified, aml_verified, sox_verified
FROM deals
WHERE status IN ('executed', 'settled', 'closed');

-- Settlement analysis
SELECT COUNT(*), SUM(amount_cents)/100 as total_usd
FROM deals
WHERE status = 'settled'
  AND settlement_date >= NOW() - INTERVAL '30 days';
```

| `remediation_notes` | TEXT | nullable | Remediation steps if failed |
| `evidence_location` | VARCHAR(500) | nullable | URI to control evidence |
| `created_at` | TIMESTAMP TZ | DEFAULT NOW() | Creation timestamp |

- `trigger_compliance_controls_updated_at` — Auto-updates `updated_at`

**Related Tables:**
- `compliance_control_mappings` — Cross-framework control equivalence

#### `compliance_control_mappings` Table

**Purpose:** Maps controls across frameworks (e.g., SOC2 CC6.1 ↔ GDPR Article 32)

**Columns:**

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | UUID | PRIMARY KEY | Auto-generated |
| `source_control_id` | UUID | NOT NULL, FK | Control in one framework |
| `target_control_id` | UUID | NOT NULL, FK | Corresponding control in another |
| `mapping_type` | VARCHAR(50) | CHECK | equivalent, related, supports, overlaps |
| `notes` | TEXT | nullable | Mapping rationale |
| `created_at` | TIMESTAMP TZ | DEFAULT NOW() | Creation timestamp |

**Constraints:**
- `source_control_id != target_control_id` — No self-mapping
- `UNIQUE(source_control_id, target_control_id)` — No duplicate mappings

**Indexes:**
```
idx_control_mappings_source              — Find outgoing mappings
idx_control_mappings_target              — Find incoming mappings
```

**Usage:**
```sql
-- Find SOC2 controls that map to GDPR Article 32
SELECT cc.control_id, cc.title, gcc.control_id as gdpr_id, gcc.title
FROM compliance_controls cc
JOIN compliance_control_mappings m ON m.source_control_id = cc.id
JOIN compliance_controls gcc ON gcc.id = m.target_control_id
WHERE cc.framework = 'SOC2'
  AND gcc.framework = 'GDPR'
  AND m.mapping_type IN ('equivalent', 'supports');
```

#### `dataset_lineage` Table

**Purpose:** Tracks data transformations and dependencies between datasets

**Columns:**

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | UUID | PRIMARY KEY | Auto-generated |
| `source_dataset_id` | UUID | NOT NULL, FK | Upstream dataset |
| `target_dataset_id` | UUID | NOT NULL, FK | Downstream dataset |
| `transformation_type` | VARCHAR(50) | CHECK | copy, aggregate, join, filter, enrich, custom |
| `transformation_description` | TEXT | nullable | What transformation occurs |
| `transformation_code` | TEXT | nullable | SQL/code for transformation |
| `last_executed_at` | TIMESTAMP TZ | nullable | Most recent execution |
| `execution_duration_ms` | INT | nullable | Execution time in milliseconds |
| `status` | VARCHAR(20) | CHECK | success, failed, pending, skipped |
| `error_message` | TEXT | nullable | Error details if failed |
| `created_at` | TIMESTAMP TZ | DEFAULT NOW() | Creation timestamp |

**Constraints:**
- `source_dataset_id != target_dataset_id` — No self-lineage
- `UNIQUE(source_dataset_id, target_dataset_id)` — No duplicate edges

**Indexes:**
```
idx_lineage_source                       — Upstream dependencies
idx_lineage_target                       — Downstream dependencies
idx_lineage_executed_at                  — Execution timeline
idx_lineage_status                       — Failed transformation tracking
```

**Usage:**
```sql
-- Find all datasets that depend on sales_orders
WITH RECURSIVE downstream AS (
  SELECT target_dataset_id, 1 as depth
  FROM dataset_lineage
  WHERE source_dataset_id = 'sales_orders_id'
  UNION ALL
  SELECT dl.target_dataset_id, ds.depth + 1
  FROM dataset_lineage dl
  JOIN downstream ds ON dl.source_dataset_id = ds.target_dataset_id
  WHERE ds.depth < 5
)
SELECT d.name, d.owner_id, downstream.depth
FROM datasets d
JOIN downstream ON d.id = downstream.target_dataset_id;
```

**Usage (Breach Detection):**
```sql
-- Find unusual access patterns (potential breach)
SELECT
  actor_id,
  COUNT(*) as access_count,
  COUNT(DISTINCT phi_record_id) as unique_records,
  MIN(action_timestamp) as first_access,
  MAX(action_timestamp) as last_access
FROM phi_audit_log
WHERE action = 'read'
  AND action_timestamp > NOW() - INTERVAL '24 hours'
GROUP BY actor_id
HAVING COUNT(*) > 100  -- Threshold for anomaly
ORDER BY access_count DESC;
```

## Performance Tuning

### Index Strategy

**Hot Paths (add indexes for):**
- Filter by status, created_by (deals)
- Filter by framework, severity (compliance_controls)
- Filter by domain, owner, sensitivity (datasets)
- Filter by patient_id_hash, consent_status (phi_records)
- Filter by action_category, timestamp, actor_id (audit_logs)

**Composite Indexes:**
- `(framework, severity)` for compliance reporting
- `(action_category, result, timestamp)` for failure analysis
- `(actor_id, timestamp)` for user timeline

### Query Optimization

```sql
-- GOOD: Uses indexes
SELECT id, name FROM deals WHERE status = 'approved' AND created_at > NOW() - INTERVAL '30 days';

-- BAD: Full table scan (no index on surname)
SELECT id, name FROM users WHERE surname = 'Smith';

-- GOOD: Aggregate with summary table
SELECT * FROM audit_log_summary WHERE summary_date = CURRENT_DATE;

-- BAD: Aggregate live (expensive)
SELECT COUNT(*), result FROM audit_logs GROUP BY result WHERE DATE(timestamp) = CURRENT_DATE;
```

## Backup & Disaster Recovery

### Full Backup Strategy

```bash
# Daily full backup (retention: 30 days)
pg_dump -Fc businessos > /backups/businessos-$(date +%Y%m%d).dump

# Incremental WAL backup (retention: 7 days)
pg_basebackup -D /wal-archive -Xstream

# Encrypted backup to S3
pg_dump businessos | gpg -e | aws s3 cp - s3://backups/businessos-$(date +%Y%m%d).dump.gpg
```

### Recovery Procedure

```bash
# Restore from full backup
pg_restore -d businessos /backups/businessos-20260326.dump

# Point-in-time recovery (use WAL archive)
pg_ctl start -D /var/lib/postgresql/data \
  -o "-c recovery_target_timeline=latest \
     -c recovery_target_time='2026-03-26 14:30:00'"
```

### Compliance Verification

```sql
-- Verify no audit logs deleted (immutable append-only)
SELECT COUNT(*) as total_logs,
       COUNT(CASE WHEN retention_until < NOW() THEN 1 END) as expired,
       MAX(created_at) as most_recent
FROM audit_logs;

-- Verify all PHI encrypted
SELECT COUNT(*) as encrypted,
       COUNT(CASE WHEN data_encrypted IS NULL THEN 1 END) as unencrypted
FROM phi_records;

-- Verify no deals without compliance checks
SELECT COUNT(*) as total,
       COUNT(CASE WHEN kcy_verified AND aml_verified AND sox_verified THEN 1 END) as fully_verified
FROM deals WHERE status IN ('executed', 'settled', 'closed');
```
